# Skill: Style HN v2.0

This skill helps you write a strong "Show HN" post: short, concrete, and
technical, with just enough context for curious developers to decide whether
to click.

The output of this skill is the full text of the Show HN submission:

- One title line.
- A short body (a few tight paragraphs).
- Plain URLs on their own lines.
- No markdown hyperlinks.

## Guiding principles

- Audience: Busy, skeptical developers scanning titles and first sentences.
- Show, do not pitch: Demonstrate what you built and why it is interesting.
- Technical first: Lead with the problem, approach, and constraints.
- Brevity: Err on the side of “uncomfortably short.”
- Candor: Be honest about limitations, tradeoffs, and what is rough.
- No fluff: Remove marketing language, emojis, and exclamation marks.

## Title

Use this pattern:

- Show HN: {what it is} – {concrete technical or usage hook}

Constraints:

- No exclamation marks.
- No hype adjectives (amazing, revolutionary, game-changing).
- Avoid vague verbs (launching, introducing, announcing).
- Prefer nouns and specific phrases over slogans.

Good examples:

- Show HN: Rocksky – Music scrobbling and discovery on the AT Protocol
- Show HN: Semble – Code search for agents that uses 98% fewer tokens than grep
- Show HN: Nibble – A tiny systems language with a 200-line compiler
- Show HN: Statewright – Visual state machines that make AI agents reliable

Weak examples:

- Show HN: My awesome new AI startup!!!
- Show HN: Introducing the future of developer productivity
- Show HN: A revolutionary platform to change how we work

If a developer cannot tell roughly what it is from the title alone, rewrite.

## Body structure

Aim for 150–300 words. Use 3–6 short paragraphs, each 1–3 sentences.
No bulleted feature lists.

Recommended structure:

1. One-line summary
2. Problem
3. Approach / what is technically interesting
4. Proof / evidence
5. Self-aware line
6. Links

### 1. One-line summary

First sentence: what it is and who it is for.

- “I built X, a {type of thing} for {who} to {do what}.”

Keep it literal and concrete.

### 2. Problem

Describe the problem in specific, measurable terms.

- “At work we spend N hours per week doing X.”
- “Existing tools either do A or B, but not both.”
- “I could not find a simple way to do Y without running Z.”

Avoid generic complaints like “developer tools are broken.”

### 3. Approach and technical details

Explain what you did that is interesting to a technical reader:

- Architecture choices.
- Algorithms or data structures.
- Constraints you optimized for (latency, memory, cost).
- Tradeoffs you knowingly accepted.

One or two short paragraphs are enough.

### 4. Proof and evidence

Give something that shows it works:

- Live demo URL.
- GitHub repository.
- Benchmarks or sample outputs.

Approximate numbers are fine if labeled as such.

### 5. Self-aware line

Include one candid, self-aware sentence:

- “This is still rough around the edges, but it is already useful to me.”
- “I am not sure if this is generally useful, but it scratched a specific itch.”
- “I almost did not post this because it feels too niche, but I would love feedback.”

This signals openness to critique.

### 6. Links

Put each URL on its own line at the end.

- Main app or demo
- Source code
- Docs or write-up (optional)

Use raw URLs, not markdown links.

## Style rules

- Voice:
  - Write like a developer who almost did not post this.
  - Prefer “I” or “we” plus concrete actions.
- Length:
  - If it feels slightly too short, it is probably right.
  - Cut anything that does not change the reader’s decision to click.
- Tone:
  - Curious, factual, and open to feedback.
  - No “launch”, “announcement”, or “go-to-market” framing.
- Content:
  - Focus on what you built and how it works.
  - Pricing, signup flows, and business models are secondary or omitted.
- Formatting:
  - Plain paragraphs only.
  - No bullet lists of features.
  - No emojis.

## Anti-patterns

Avoid:

- “We built X to solve Y for everyone.”
- “Excited to share our launch of…”
- Long origin stories.
- Roadmaps, hiring pitches, investor language.
- Feature laundry lists.
- Over-explaining obvious things.
- Defensive disclaimers (“please be nice”).

If a sentence would fit in a press release, delete it.

## Templates

### Quick template

Title:

- Show HN: {product} – {concrete technical or usage hook}

Body:

- Sentence 1: “I built {product}, a {type of thing} for {who} to {do what}.”
- Paragraph 2: One short paragraph on the problem.
- Paragraph 3: One short paragraph on the approach.
- Paragraph 4: One short paragraph with proof.
- Sentence: One self-aware line.
- Links: Each URL on its own line.

### Example 1 (developer tool)

Title:

- Show HN: Tracecut – Minimal request tracing for small services

Body:

I built Tracecut, a tiny request tracing tool for small services that do not
justify a full observability stack.

Most tracing tools assume you have a service mesh or a big budget. For my side
projects, I just wanted to see how a single request moved through a few HTTP
handlers and background jobs without running a separate cluster.

Tracecut is a single Go library that injects a trace ID into your handlers and
propagates it through context. It writes spans to a local SQLite database and
serves a simple HTML view. A typical trace is a few kilobytes.

On my small services, Tracecut adds around 1–2 ms per request and uses less
than 50 MB of RAM on a small VPS.

I almost did not post this because it feels too small, but it has already
saved me hours debugging weird edge cases.

https://tracecut.example.com
https://github.com/example/tracecut

### Example 2 (consumer-ish app)

Title:

- Show HN: GridTravel – Share and remix travel routes as simple grids

Body:

I built GridTravel, a small web app for sharing travel routes as simple
day-by-day grids instead of long blog posts or messy docs.

When I plan trips with friends, we end up with spreadsheets, chat threads, and
random links. It is hard to see the whole trip at a glance or reuse someone
else’s itinerary.

GridTravel lets you build a trip as a grid of days and time blocks. Each cell
can hold a place, a note, or a link. Under the hood it is just JSON, so you
can export or version it in Git. No login required; trips are stored as signed
URLs.

Right now it is missing collaboration and mobile polish, but a few friends
have already used it to plan real trips.

https://gridtravel.app
https://github.com/example/gridtravel

## How to use this skill

When asked to draft or revise a Show HN post:

1. Identify:
   - What it is.
   - Who it is for.
   - The concrete problem.
   - The technically interesting parts.
   - Any proof (usage, benchmarks, demos).
2. Generate:
   - A title following the “Show HN: … – …” pattern.
   - A 150–300 word body using the structure above.
   - A self-aware line.
   - Plain URLs on separate lines.
3. Remove:
   - Hype, marketing phrases, and feature lists.
   - Anything that does not help a skeptical developer decide to click.
